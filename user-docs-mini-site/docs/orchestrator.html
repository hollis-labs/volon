Orchestrator Mode

The primary session role: reads state, selects work, delegates, writes results, and loops.

Orchestrator mode is a process role, not a new architecture.

Purpose

Define a thin primary session that runs Forge by reading bootstrap/PCC/tasks/config, selecting the next unit of work, delegating bounded work to sub-agents (optional), applying changes as the single writer, verifying, committing per policy, logging, and updating bootstrap.

Roles

Orchestrator default role
Responsibilities:
  • + Interpret ground truth from repo artifacts
  • + Decide what to do next
  • + Enforce policies (single-writer, limits, verification, commits)
  • + Delegate bounded work to sub-agents (optional)
  • + Integrate results and write changes
Must not:
  • — Offload state writes to sub-agents
  • — Allow recursive sub-agent spawning
Worker bounded sub-agent
Execute a single scoped objective and return results in the requested format. Must not edit files, update tasks/logs/PCC/bootstrap, or spawn agents.
Reviewer / Investigator bounded sub-agent
Same constraints as Worker. Intended for scans (security, dead code), summaries (knowledge artifacts), and generating options or alternatives.

Single-writer rule

Only the Orchestrator may write or update these paths. All sub-agents are read-only.

.forge/tasks/**
.forge/backlog/**
.forge/logs/**
.forge/pcc/**
.forge/bootstrap.md
.forge/bootstrap/history/**

When to delegate

Delegate when
  • + Reading multiple filesystem areas
  • + Summarizing code sections
  • + Scanning diffs
  • + Generating option lists
Do not delegate
  • — File edits, renames, or refactors
  • — Tasks requiring coherent multi-step reasoning
  • — Anything that writes to state files

Orchestrator loop (canonical)

1 Read .forge/bootstrap.md if present.
2 Determine next action: pick next todo task OR next workflow step.
3 Optional: delegate bounded analysis to sub-agents.
4 Apply changes locally (single writer).
5 Verify against acceptance criteria.
6 Commit per policy (task vs iteration mode).
7 Update task status and append to Updates section.
8 Write run log.
9 Finalize: /bootstrap-update

Pausing and resuming

"Pause" is implemented by externalizing state to disk, not suspending memory. A paused session can be resumed deterministically in a fresh context because all state lives in files.

/pause-task [mode] [note]

Writes task status, bootstrap resume point, optional run log. Preferred mode: restart.

/resume-task [note]

Re-grounds from bootstrap. Identifies paused task by paused_task_id in bootstrap frontmatter. Continues loop.

Why restart is preferred: Agents cannot reliably clear context in-place. Restarting a session is deterministic, testable, and matches the behavior of scheduled/automated runs.
